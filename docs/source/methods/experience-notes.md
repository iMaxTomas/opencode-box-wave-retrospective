# Heavy Task Experience Notes

## Purpose

这份文件保存当前仓库里值得长期复用的重任务经验。

它不是某个任务的即时说明。

它也不是 OpenSpec 全局硬规范。

它只是：

- 当前提炼出来的高价值经验
- 以后可以继续增删和修正的方法笔记

## 当前最值得保留的 5 条经验

### 1. 先分清“运行健康”和“基线健康”

不要把这两件事混成一件事。

- 运行健康：
  - 现在能不能跑
  - 现在能不能继续用
- 基线健康：
  - source-of-truth 清不清楚
  - worktree 干不干净
  - 当前状态能不能当长期基线

很多复杂任务里：

- 系统是能用的
- 但基线是不健康的

这两者必须分开判断。

### 2. 重任务必须同时摆出三条证据线

重任务不能只写执行步骤。

至少要并排摆出：

- 设计诉求
- 反驳 / 风险
- 现场事实

这样后面的 repair 才不是拍脑袋。

### 3. 先找最影响后续动作的那个模糊点

不要先修最显眼的症状。

先找：

- 哪个歧义最影响后面所有动作
- 哪个问题不先定，后面的每一步都会再次漂

这个点，往往比最吵的症状更值得先处理。

### 4. 先做波次菜单，再只开一个波次

重任务不能一口气把所有相关问题一起动。

正确做法是：

1. 先列出后续波次菜单
2. 每个波次只解决一种问题
3. 当前窗口只开其中一个

也就是：

先画修复地图，再只开一条路。

### 5. sidecar 负责收证据，主线程负责定修复方向

在重任务里：

- sidecar 最适合并行收证据
- 主线程最适合统一判断方向

不要让 sidecar 接管主修复方向。

也不要让主线程一边修，一边自己重新查完所有证据。

## 这 5 条真正帮助的是什么

它们不是格式要求。

它们帮助我：

- 我怎么判断复杂任务
- 我怎么避免一轮发散
- 我怎么把“能用”和“可长期维护”分开
- 我怎么决定先做哪一波

## 适用场景

这 5 条尤其适合：

- 一轮里已经不只是单点修复
- 任务同时涉及：
  - 运行面
  - source-of-truth
  - 策略面
  - 长期治理面
- 用户明显已经不想再看“想到哪修到哪”的推进方式

## 不该怎么用

不要把这 5 条抬成“核心规则”。

也不要把它们直接塞进每个任务卡里原样复读。

正确用法是：

- 经验留在这里
- 每个具体 change 只引用并落具体版本

一句话：

这里保存的是“当前重任务经验”，不是“每轮作战详情”。
